# RAG Algo Solver — Обзор проекта

## Что это и зачем

RAG Algo Solver — веб-приложение, которое помогает пользователям решать алгоритмические задачи с помощью большой языковой модели, усиленной базой знаний (подход Retrieval-Augmented Generation). Пользователь описывает задачу на естественном языке, а система находит семантически похожие задачи с разборами в своей базе, формирует из них контекст и генерирует осмысленный ответ — не «с нуля», а опираясь на накопленный опыт реальных решений.

Целевая аудитория — студенты и участники олимпиад, которым нужен не просто ответ, а подсказка с объяснением ключевой идеи, подкреплённая разборами аналогичных задач.

---

## Что видит пользователь

Интерфейс минималистичен и выполнен в терминальном стиле. Взаимодействие сводится к нескольким шагам:

1. **Регистрация и вход.** Классическая пара логин/пароль, аутентификация через JWT-токен.
2. **Ввод запроса.** Пользователь описывает алгоритмическую задачу в свободной форме.
3. **Ожидание обработки.** Интерфейс показывает текущий статус — обогащение запроса, поиск похожих задач, формирование контекста, генерация ответа. Фронтенд опрашивает бэкенд каждые 3 секунды.
4. **Получение результата.** Ответ отображается с эффектом печатной машинки — это краткое объяснение ключевой идеи решения. Рядом — список похожих задач из базы знаний со ссылками на оригиналы и их разборы.
5. **История запросов.** Все предыдущие запросы доступны в боковой панели с пагинацией.

Администратор может управлять базой знаний — добавлять, редактировать и удалять задачи через отдельные API-эндпоинты. Любое изменение задачи через API автоматически синхронизируется с векторной базой, поэтому новая задача сразу становится доступной для поиска.

## Продуктовые решения

**RAG вместо чистой генерации.** Языковая модель сама по себе может «галлюцинировать» — придумывать несуществующие алгоритмы или давать некорректные решения. Подход RAG снижает эту проблему: модель получает контекст из реальных задач с проверенными разборами, что повышает точность и полезность ответа.

**Объяснение идеи, а не готовый код.** Система намеренно настроена давать не полное решение с кодом, а краткое объяснение сути подхода: какой алгоритм применить и почему он работает для этой задачи. Это соответствует учебной цели — подтолкнуть пользователя к решению, а не выдать его за него.

**Канонизация запроса (симметрия с базой знаний).** Условия задач часто обёрнуты в художественный сюжет («Вася собирает монеты…»), который мешает семантическому поиску: две одинаковые по сути задачи с разными историями дают далёкие эмбеддинги. Поэтому и пользовательский запрос, и задачи в базе проходят одинаковую LLM-обработку — приведение к абстрактному, «обезличенному» виду (массивы, графы, ограничения), без нарратива. Это ключевое решение: благодаря симметрии запрос и задачи лежат в одном смысловом пространстве, и поиск находит действительно похожие задачи.

**Асинхронная обработка.** Запрос пользователя не блокирует HTTP-соединение. Он проходит через конвейер из четырёх этапов, каждый из которых — отдельный шаг в очереди сообщений. Пользователь видит прогресс в реальном времени, а не просто ждёт у пустого экрана.

**Пополняемая база знаний из двух источников.** В комплекте идут парсеры задач с платформ **e-olymp.com** и **AtCoder**, которые автоматически собирают условия задач и разборы (эдиториалы). Собранные задачи проходят офлайн-обогащение той же LLM-канонизацией, что и запросы пользователя, и загружаются в сервис. На текущий момент в базе знаний — порядка 500 задач с двух платформ. База расширяема: можно подключать новые платформы и добавлять задачи вручную через API.

**LLM и эмбеддинги от Yandex Cloud.** В качестве генеративной модели и модели эмбеддингов используется Yandex Cloud (API, совместимый с OpenAI), что обеспечивает хорошее качество работы на русском языке и работу в рамках российского облачного контура.

## Архитектурные решения

### Чистая слоёная архитектура

Проект разделён на три слоя с чётким направлением зависимостей (`presentation → infrastructure → domain`):

- **Domain** — ядро системы. Содержит бизнес-модели (запрос, задача, ответ), абстрактные интерфейсы сервисов и репозиториев. Не зависит ни от фреймворков, ни от базы данных, ни от конкретного LLM-провайдера. Это позволяет подменять реализации без изменения бизнес-логики.

- **Infrastructure** — реализации абстракций из Domain. Здесь живут SQL-репозитории (PostgreSQL через async SQLAlchemy), векторный репозиторий и поиск (Qdrant), клиенты Yandex Cloud для генерации и эмбеддингов, публикация сообщений в Kafka, а также адаптеры для обогащения запросов, сборки контекста и композиции промптов.

- **Presentation** — точки входа: REST API на FastAPI и Kafka-обработчики на FastStream. Этот слой принимает запросы от пользователя и оркестрирует вызовы инфраструктурного слоя.

Такая структура делает каждый компонент заменяемым. Нужно сменить LLM-провайдера? Пишется новая реализация `LLMClient`, остальной код не затрагивается. Нужна другая модель эмбеддингов? Реализуется новый `EmbeddingClient` (в проекте, помимо облачного, есть и локальная реализация на BGE-M3).

### Асинхронный конвейер через Kafka

Обработка запроса разбита на четыре последовательных этапа, связанных через топики Kafka:

```
query.enrich → query.search → query.compose → query.generate
```

Каждый этап — независимый обработчик, который:
- читает текущее состояние из базы данных,
- выполняет свою часть работы,
- сохраняет результат обратно в БД,
- публикует сообщение в следующий топик.

Содержательно этапы делают следующее:
- **enrich** — приводит текст запроса к канонической форме через LLM;
- **search** — строит эмбеддинг канонизированного запроса и ищет в Qdrant ближайшие задачи;
- **compose** — собирает из найденных задач контекст и формирует финальный промпт;
- **generate** — отправляет промпт в LLM и сохраняет ответ.

Статус запроса отражает текущий этап: `pending → enriching → searching → composing → generating → completed` (или `failed`).

Такой подход даёт несколько преимуществ:
- **Отказоустойчивость.** Падение одного этапа не теряет данные — сообщение остаётся в Kafka, а состояние зафиксировано в БД.
- **Масштабируемость.** Каждый этап можно масштабировать независимо (например, запустить несколько экземпляров генератора).
- **Наблюдаемость.** Всегда видно, на каком этапе находится запрос и где произошёл сбой (статус и сообщение об ошибке хранятся в БД).

### Dependency Injection через контейнер

Все зависимости (репозитории, клиенты, адаптеры) создаются и связываются в одном месте — `AppContainer` на базе библиотеки `dependency-injector`. Это единственная точка конфигурации: ни один модуль не создаёт свои зависимости напрямую. Смена реализации (например, облачных эмбеддингов на локальные) — это изменение одной строки в контейнере.

### Абстракции конвейерных шагов

Каждый этап обработки представлен абстрактным интерфейсом, за которым стоит конкретная стратегия. Это позволяет комбинировать и заменять реализации без изменения остального кода:

| Абстракция | Назначение | Текущая реализация |
|---|---|---|
| `QueryEnricher` | Канонизация текста запроса | LLM-обогатитель (убирает сюжет, оставляет суть) |
| `EmbeddingClient` | Преобразование текста в вектор | Yandex Cloud (есть и локальная на BGE-M3) |
| `SimilarTaskSearcher` | Поиск похожих задач | Векторный поиск в Qdrant по косинусной близости |
| `TaskContextBuilder` | Сборка контекста из найденных задач | Конкатенация условий и разборов |
| `PromptComposer` | Формирование финального промпта | Промпт «объясни ключевую идею» |
| `LLMClient` | Генерация ответа | Yandex Cloud API |
| `VectorTaskRepository` | Синхронизация задач с векторной БД | Upsert/delete точек в Qdrant |

Для обогатителя предусмотрена и более гибкая схема — конвейер обогащения из независимых блоков (`EnrichmentPipeline` / `EnrichmentBlock`), которые можно выстраивать в цепочку. Это задел под более сложную предобработку запроса (например, выделение алгоритмических паттернов отдельным шагом).

### Семантический поиск

Поиск похожих задач опирается на векторное представление смысла, а не на совпадение слов:

1. Канонизированный текст запроса (и каждой задачи при загрузке) превращается в эмбеддинг моделью Yandex Cloud.
2. Векторы задач хранятся в Qdrant вместе с полезной нагрузкой (идентификатор, заголовок, текст, разбор).
3. По запросу Qdrant возвращает несколько ближайших по косинусной мере задач, из которых строится контекст.

Симметрия обработки (одинаковая канонизация запроса и задач) — необходимое условие, чтобы эта близость была осмысленной.

### Схема данных

Пять таблиц, связанных логически:

- **users** — учётные записи с хешированными паролями и флагом администратора.
- **tasks** — база знаний: заголовок, условие, разбор (`solution`), ссылки на источник и на разбор, комментарий.
- **queries** — пользовательские запросы с итоговым ответом.
- **query_processing** — промежуточное состояние конвейера (1:1 с запросом): исходный и обогащённый текст, собранный контекст, статус, сообщение об ошибке.
- **query_similar_tasks** — связь «многие ко многим» между запросами и найденными похожими задачами.

Разделение `queries` и `query_processing` — осознанный выбор: пользовательский запрос и его ответ хранятся отдельно от технического состояния конвейера, что упрощает API и не засоряет пользовательские данные служебной информацией. Задачи при этом хранятся в двух местах одновременно: реляционно в PostgreSQL (источник истины для CRUD и отображения) и векторно в Qdrant (для поиска); согласованность обеспечивает `VectorTaskRepository`, вызываемый при создании, изменении и удалении задачи.

## Сбор и подготовка базы знаний

База знаний наполняется офлайн-конвейером из трёх шагов, отдельным для каждой платформы:

1. **Парсинг.** Парсеры `eolymp_parser` и `atcoder_parser` собирают со страниц условия задач и текстовые разборы. Задачи без разбора пропускаются; видео-разборы и код решений в эдиториалах не сохраняются. Результат — JSON-файл на каждую задачу.
2. **Обогащение (enrichment).** Скрипты обогащения прогоняют условия через ту же LLM-канонизацию, что и запросы пользователя, добавляя «обезличенную» версию условия. Именно она используется для построения эмбеддинга — чтобы поиск опирался на суть задачи, а не на сюжет.
3. **Загрузка.** Скрипты загрузки отправляют обогащённые задачи в сервис через `POST /tasks` под учётной записью администратора. Загрузка идемпотентна: ключи уже отправленных задач сохраняются в файле состояния и при повторном запуске пропускаются.

Такое разделение «парсинг → обогащение → загрузка» позволяет собирать и дочищать базу частями, переживать обрывы и подключать новые платформы по единому шаблону.

## Инфраструктура

Вся инфраструктура поднимается через Docker Compose:

- **PostgreSQL** — основное реляционное хранилище данных, миграции через Alembic.
- **Apache Kafka** — очередь сообщений для конвейера обработки.
- **Qdrant** — векторная база данных для семантического поиска похожих задач.
- **Kafbat UI** — веб-интерфейс для мониторинга топиков Kafka.

Бэкенд состоит из двух процессов: API-сервер (FastAPI) и потребитель Kafka (FastStream). Фронтенд — Express-сервер, раздающий статику и проксирующий API-запросы на бэкенд. Без фронтенд-фреймворков, на чистом JavaScript — простота без накладных расходов.

Конфигурация вынесена в Pydantic Settings-классы по группам (`PostgresConfig`, `KafkaConfig`, `JWTConfig`, `YandexCloudConfig`, `QdrantConfig`) и загружается из окружения при старте.

## Текущее состояние и направление развития

Система функционально завершена по основному сценарию: работают REST API, аутентификация, Kafka-конвейер, LLM-канонизация запроса, векторный поиск в Qdrant, интеграция с LLM для генерации и фронтенд. База знаний наполнена порядка 500 задачами с двух платформ.

Ключевые точки роста:

- **Качество поиска и обогащения.** Тонкая настройка канонизации и числа возвращаемых задач, эксперименты с моделями эмбеддингов (включая локальную BGE-M3), оценка релевантности выдачи.
- **Более сложная предобработка запроса.** Реализация дополнительных блоков обогащения (например, выделение алгоритмических паттернов — графы, динамическое программирование, жадные алгоритмы) в рамках уже существующего конвейера `EnrichmentPipeline`.
- **Расширение базы знаний.** Подключение новых платформ (Codeforces, LeetCode) по тому же шаблону «парсинг → обогащение → загрузка».

Система спроектирована так, что каждое из этих улучшений — это реализация уже существующего интерфейса или добавление блока в конвейер, без перестройки архитектуры.
