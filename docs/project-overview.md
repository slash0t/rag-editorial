# RAG Algo Solver — Обзор проекта

## Что это и зачем

RAG Algo Solver — веб-приложение, которое помогает пользователям решать алгоритмические задачи с помощью большой языковой модели, усиленной базой знаний (подход Retrieval-Augmented Generation). Пользователь описывает задачу на естественном языке, а система находит похожие задачи с решениями в своей базе, формирует контекст и генерирует осмысленный ответ — не «с нуля», а опираясь на накопленный опыт.

Целевая аудитория — студенты и участники олимпиад, которым нужен не просто ответ, а подсказка с объяснением, подкреплённая реальными решениями аналогичных задач.

---

## Что видит пользователь

Интерфейс минималистичен и выполнен в терминальном стиле. Взаимодействие сводится к нескольким шагам:

1. **Регистрация и вход.** Классическая пара логин/пароль, аутентификация через JWT-токен.
2. **Ввод запроса.** Пользователь описывает алгоритмическую задачу в свободной форме.
3. **Ожидание обработки.** Интерфейс показывает текущий статус — обогащение запроса, поиск похожих задач, формирование контекста, генерация ответа. Фронтенд опрашивает бэкенд каждые 3 секунды.
4. **Получение результата.** Ответ отображается с эффектом печатной машинки. Рядом — список похожих задач из базы знаний со ссылками на оригиналы и их решения.
5. **История запросов.** Все предыдущие запросы доступны в боковой панели с пагинацией.

Администратор может управлять базой знаний — добавлять, редактировать и удалять задачи через отдельные API-эндпоинты.

## Продуктовые решения

**RAG вместо чистой генерации.** Языковая модель сама по себе может «галлюцинировать» — придумывать несуществующие алгоритмы или давать некорректные решения. Подход RAG снижает эту проблему: модель получает контекст из реальных задач с проверенными решениями, что повышает точность и полезность ответа.

**Асинхронная обработка.** Запрос пользователя не блокирует HTTP-соединение. Он проходит через конвейер из четырёх этапов, каждый из которых — отдельный шаг в очереди сообщений. Пользователь видит прогресс в реальном времени, а не просто ждёт у пустого экрана.

**Пополняемая база знаний.** В комплекте идёт парсер задач с платформы e-olymp.com, который автоматически собирает условия задач и разборы (эдиториалы). На текущий момент собрано 174+ задач. База расширяема — администратор может добавлять задачи и вручную.

**LLM от Yandex Cloud.** В качестве генеративной модели используется Yandex Cloud (совместимый с OpenAI API), что обеспечивает хорошее качество генерации на русском языке и работу в рамках российского облачного контура.

## Архитектурные решения

### Чистая слоёная архитектура

Проект разделён на три слоя с чётким направлением зависимостей:

- **Domain** — ядро системы. Содержит бизнес-модели (запрос, задача, ответ), абстрактные интерфейсы сервисов и репозиториев. Не зависит ни от фреймворков, ни от базы данных, ни от конкретного LLM-провайдера. Это позволяет подменять реализации без изменения бизнес-логики.

- **Infrastructure** — реализации абстракций из Domain. Здесь живут SQL-репозитории (PostgreSQL через async SQLAlchemy), клиент Yandex Cloud LLM, публикация сообщений в Kafka, а также адаптеры для обогащения запросов, поиска и композиции промптов.

- **Presentation** — точки входа: REST API на FastAPI и Kafka-обработчики на FastStream. Этот слой принимает запросы от пользователя и оркестрирует вызовы инфраструктурного слоя.

Такая структура делает каждый компонент заменяемым. Нужно сменить LLM-провайдера? Пишется новая реализация `LLMClient`, остальной код не затрагивается. Нужно другое хранилище? Реализуется новый репозиторий.

### Асинхронный конвейер через Kafka

Обработка запроса разбита на четыре последовательных этапа, связанных через топики Kafka:

```
query.enrich → query.search → query.compose → query.generate
```

Каждый этап — независимый обработчик, который:
- читает текущее состояние из базы данных,
- выполняет свою часть работы,
- сохраняет результат обратно в БД,
- публикует сообщение в следующий топик.

Статус запроса отражает текущий этап: `pending → enriching → searching → composing → generating → completed` (или `failed`).

Такой подход даёт несколько преимуществ:
- **Отказоустойчивость.** Падение одного этапа не теряет данные — сообщение остаётся в Kafka.
- **Масштабируемость.** Каждый этап можно масштабировать независимо (например, запустить несколько экземпляров генератора).
- **Наблюдаемость.** Всегда видно, на каком этапе находится запрос и где произошёл сбой.

### Dependency Injection через контейнер

Все зависимости (репозитории, клиенты, адаптеры) создаются и связываются в одном месте — `AppContainer` на базе библиотеки `dependency-injector`. Это единственная точка конфигурации: ни один модуль не создаёт свои зависимости напрямую. Смена реализации (например, с «заглушки» на полноценный Qdrant-поиск) — это изменение одной строки в контейнере.

### Абстракции конвейерных шагов

Каждый этап обработки представлен абстрактным интерфейсом:

| Абстракция | Назначение | Текущая реализация |
|---|---|---|
| `QueryEnricher` | Обогащение текста запроса | Passthrough (без изменений) |
| `SimilarTaskSearcher` | Поиск похожих задач | Заглушка (возвращает пустой список) |
| `TaskContextBuilder` | Сборка контекста из найденных задач | Конкатенация текстов |
| `PromptComposer` | Формирование финального промпта | Шаблон с подстановкой |
| `LLMClient` | Генерация ответа | Yandex Cloud API |

Это «стратегии», которые можно комбинировать и заменять. Например, `QueryEnricher` может быть расширен NLP-обработкой для выделения ключевых алгоритмических концепций из запроса.

### Схема данных

Пять таблиц, связанных логически:

- **users** — учётные записи с хешированными паролями и флагом администратора.
- **tasks** — база знаний: условия задач, решения, ссылки на источники.
- **queries** — пользовательские запросы с итоговым ответом.
- **query_processing** — промежуточное состояние конвейера (1:1 с запросом): обогащённый текст, контекст, статус, сообщение об ошибке.
- **query_similar_tasks** — связь «многие ко многим» между запросами и найденными похожими задачами.

Разделение `queries` и `query_processing` — осознанный выбор: пользовательский запрос и его ответ хранятся отдельно от технического состояния конвейера, что упрощает API и не засоряет пользовательские данные служебной информацией.

## Инфраструктура

Вся инфраструктура поднимается через Docker Compose:

- **PostgreSQL** — основное хранилище данных, миграции через Alembic.
- **Apache Kafka** — очередь сообщений для конвейера обработки.
- **Qdrant** — векторная база данных для семантического поиска похожих задач.
- **Kafbat UI** — веб-интерфейс для мониторинга топиков Kafka.

Фронтенд — Express-сервер, раздающий статику и проксирующий API-запросы на бэкенд. Без фреймворков, на чистом JavaScript — простота без накладных расходов.

## Текущее состояние и направление развития

Архитектурный каркас полностью готов: работает REST API, аутентификация, Kafka-конвейер, интеграция с LLM и фронтенд. Ключевые точки роста:

- **Интеграция Qdrant** — подключение векторного поиска, чтобы этап `search` находил действительно похожие задачи по семантической близости, а не возвращал пустой результат.
- **Умное обогащение запросов** — замена passthrough-обогатителя на NLP-модуль, который выделяет алгоритмические паттерны (графы, динамическое программирование, жадные алгоритмы) для более точного поиска.
- **Расширение базы знаний** — парсер e-olymp уже собрал 174+ задач, но потенциально можно подключить другие платформы (Codeforces, LeetCode).

Система спроектирована так, что каждое из этих улучшений — это реализация уже существующего интерфейса, без перестройки архитектуры.
